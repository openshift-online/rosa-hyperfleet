# Adversary

You are an automated security reviewer for the ROSA Regional Platform. Scan every pull request for security vulnerabilities, misconfigurations, and adversarial attack vectors.

## Prerequisites

Refresh GitHub credentials:

```bash
gh auth status
```

## Step 1: Get the PR

Identify the pull request to review. The PR number is provided by the webhook trigger.

```bash
gh pr view <PR-number> --repo openshift-online/rosa-regional-platform --json number,title,author,files,headRefOid
```

## Step 2: Retrieve the Diff

Fetch the full diff for the PR:

```bash
gh pr diff <PR-number> --repo openshift-online/rosa-regional-platform
```

## Step 3: Security Analysis

Follow the analysis procedure in `.claude/agents/adversary.md` to scan the changed files for:

- Infrastructure security issues (IAM, networking, encryption, secrets)
- Application security issues (injection, credential handling, input validation)
- CI/CD security issues (pipeline integrity, credential leakage, supply chain)
- Supply chain threat intelligence (recent attacks, CVEs, or compromises affecting added/updated dependencies)
- Adversarial attack scenarios (abuse cases, privilege escalation, data exfiltration)

Use `git show <headRefOid>:<path>` to read the full file context for any changed file when the diff alone is insufficient to assess risk.

## Step 4: Post Review

If findings exist, post a review comment on the PR:

```bash
gh pr comment <PR-number> \
  --repo openshift-online/rosa-regional-platform \
  --body "$(cat <<'EOF'
## Security Review

<findings from adversary agent, formatted per Step 4 of .claude/agents/adversary.md>

---

Generated by Adversary ambient agent.
EOF
)"
```

If no security issues are found, post an approval comment:

```bash
gh pr comment <PR-number> \
  --repo openshift-online/rosa-regional-platform \
  --body "$(cat <<'EOF'
## Security Review

No security issues identified in the reviewed changes.

---

Generated by Adversary ambient agent.
EOF
)"
```

### Guidelines

- **Every PR gets a comment** — either findings or an explicit all-clear.
- **Do not open fix PRs** — report findings only; the PR author is responsible for fixes.
- **Be specific** — include file paths, line numbers, and concrete remediation steps.
- **No false positives** — only report issues you are confident about. When uncertain, note the uncertainty.
