Findings

▶ HIGH No VPC Flow Logs Enabled
File
terraform/modules/vpc/main.tf
Category
Infrastructure — Network Monitoring
Issue
The VPC module defines VPC endpoints and security groups but has no aws_flow_log resource. VPC flow logs are essential for network traffic visibility, intrusion detection, and incident investigation.
Impact
Without flow logs, there is no record of accepted/rejected network connections. An attacker performing lateral movement or data exfiltration within the VPC would leave no network-level audit trail. This is also a gap for FedRAMP AU-12 (Audit Generation) which requires capturing network traffic events.
Remediation
  1. Add aws_flow_log resource to terraform/modules/vpc/main.tf sending to CloudWatch Logs (KMS-encrypted, matching the project’s pattern for other log groups).
  2. Verify flow logs appear in CloudWatch after apply.
  3. Add a CI check or Terraform sentinel policy requiring aws_flow_log in VPC modules.
  4. Consider enabling flow logs at the subnet level for finer-grained visibility. Gate behind enable_vpc_flow_logs flag defaulting to true.
▶ HIGH No Kubernetes NetworkPolicies
File
argocd/config/ (all charts)
Category
Kubernetes — Network Segmentation
Issue
Zero NetworkPolicy resources found across all ArgoCD Helm chart configurations for both Regional Clusters and Management Clusters. Without NetworkPolicies, any pod can communicate with any other pod unrestricted.
Impact
If any pod is compromised, the attacker has unrestricted network access to all other pods including control plane services (Maestro, CLM, ArgoCD), databases, and internal APIs. This violates the principle of least privilege at the network layer.
Remediation
  1. Define default-deny NetworkPolicies for each namespace, then add allow rules for required traffic flows.
  2. Start with monitoring/audit-only mode using Calico or Cilium network policy logging before enforcing.
  3. Add NetworkPolicy templates to critical charts: maestro-server, platform-api, hyperfleet-api-chart, hyperfleet-sentinel-chart.
  4. Test with kubectl describe networkpolicy and connectivity tests between pods.
▶ HIGH CI Container Image Runs as Root
File
ci/Containerfile
Category
Containers — Privilege
Issue
The CI build-root image has no USER instruction. All CI tasks (Terraform, Helm, tests) execute as UID 0 inside the container.
Impact
If a CI pipeline is compromised (e.g., through a malicious PR that escapes the build sandbox), the attacker has root inside the container, maximizing the blast radius. Combined with awscurl and AWS CLI tools, this could lead to credential theft.
Remediation
  1. Add USER 65534 at the end of the Containerfile (matching platform-image/Dockerfile).
  2. Verify CI jobs still work — the existing chmod 777 cache directories should accommodate this.
  3. If root is required for specific operations, use a multi-stage build.
▶ MEDIUM Unpinned Container Base Images
File
ci/Containerfile:5, images/zoa-tools/Dockerfile:1
Category
Supply Chain — Container Images
Issue
Two of three Dockerfiles use :latest tags for base images. The third (platform-image/Dockerfile) correctly pins to ubi9/ubi-minimal:9.7-1770267347.
Impact
A compromised or broken :latest tag could silently introduce vulnerabilities. For the CI image, this affects every PR check. For zoa-tools, this affects the Trusted Actions execution environment.
Remediation
  1. Pin both images to specific digest or version tags.
  2. Set up Dependabot or Renovate for automated update PRs.
▶ MEDIUM Curl-Pipe-Bash for Helm Installation
File
ci/Containerfile:85-86
Category
Supply Chain — Build Pipeline
Issue
Helm is installed by piping a script from raw.githubusercontent.com directly to bash. While the version is pinned and VERIFY_CHECKSUM=true is set, the script itself is fetched from a tag reference, not a commit hash.
Impact
A supply chain attack on the Helm repository could inject malicious code into the CI build image. The checksum verification of the binary mitigates the binary itself, but the installer script runs before that verification.
Remediation
  1. Pin the script URL to a commit hash instead of a version tag.
  2. Alternatively, download the Helm tarball directly with explicit checksum verification.
▶ MEDIUM Missing Pod Security Contexts (22+ Charts)
Files
argocd/config/regional-cluster/platform-api/values.yaml, maestro-server/values.yaml, hyperfleet-api-chart/values.yaml, shared/argocd/values.yaml, and 18+ others
Category
Kubernetes — Pod Security
Issue
22+ Helm chart values.yaml files have no explicit securityContext settings. Only 4 charts (cloudwatch-exporter, grafana, vector, external-secrets) set proper contexts.
Impact
Pods may run as root, with writable root filesystems, with all Linux capabilities, and with privilege escalation allowed — depending on namespace PSS level.
Remediation
  1. Add baseline security context to all charts: 
    securityContext:
  runAsNonRoot: true
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop: [ALL]
  2. Prioritize: platform-api, maestro-server, hyperfleet-api-chart, hyperfleet-sentinel-chart.
  3. Enforce via namespace-level Pod Security Standards (Restricted).
▶ MEDIUM Vector readOnlyRootFilesystem Disabled
File
argocd/config/regional-cluster/vector/values.yaml:34, argocd/config/management-cluster/vector/values.yaml:63
Category
Kubernetes — Pod Security
Issue
Vector DaemonSet containers run with readOnlyRootFilesystem: false. Vector runs on every node, amplifying the impact.
Impact
A writable root filesystem allows an attacker to persist malware, modify binaries, or install additional tools.
Remediation
  1. Set readOnlyRootFilesystem: true and add emptyDir volume mounts for data/cache directories.
  2. Test log collection still works.
  3. Align RC and MC Vector configurations.
▶ MEDIUM Terraform Provider Version Constraints Lack Upper Bounds
Files
terraform/config/pipeline-management-cluster/versions.tf, pipeline-regional-cluster/versions.tf, dns-environment-zone/versions.tf, and others
Category
Supply Chain — Infrastructure
Issue
All provider constraints use >= floor-only syntax (e.g., version = ">= 6.0"). Lock files (.terraform.lock.hcl) are gitignored.
Impact
A compromised or breaking provider version could be silently adopted on next terraform init.
Remediation
  1. Add upper-bound constraints: version = ">= 6.0, < 7.0".
  2. Alternatively, commit .terraform.lock.hcl files (remove from .gitignore).
  3. Use Renovate or Dependabot for controlled upgrades.
▶ MEDIUM Security Group Egress Unrestricted to 0.0.0.0/0
Files
terraform/modules/maestro-infrastructure/rds.tf:122-128, bastion/main.tf:89-94, ecs-bootstrap/security-groups.tf:11-18, hyperfleet-infrastructure/amazonmq.tf:113-116, hyperfleet-infrastructure/rds.tf:122-125
Category
Infrastructure — Network Security
Issue
Five security groups have egress rules allowing all protocols to 0.0.0.0/0. While ingress rules are properly scoped, unrestricted egress allows compromised resources to communicate with arbitrary destinations.
Impact
A compromised RDS instance, bastion, or ECS task could exfiltrate data to attacker-controlled endpoints.
Remediation
  1. For RDS: remove egress or restrict to VPC CIDR only.
  2. For ECS/bastion: restrict egress to specific CIDR ranges or use VPC endpoints.
  3. Consider VPC endpoint policies to further restrict egress.
▶ MEDIUM Binary Downloads Without Checksum Verification
Files
images/zoa-tools/Dockerfile:28-45, terraform/modules/platform-image/Dockerfile:36-63
Category
Supply Chain — Container Build
Issue
Multiple binary tools (kubectl, oc, AWS CLI, yq, k9s, stern) are downloaded via curl without SHA256 checksum verification. The CI Containerfile correctly verifies Terraform, k6, and promtool.
Impact
MITM or compromised mirrors could serve malicious binaries. Especially concerning for zoa-tools (production Trusted Actions image).
Remediation
  1. Add checksum verification for all binary downloads following the CI Containerfile pattern.
  2. Prioritize zoa-tools/Dockerfile.
▶ MEDIUM Unpinned Python Dependencies in CI
File
ci/Containerfile:196-198
Category
Supply Chain — Dependencies
Issue
awscurl and pyyaml installed without version pins via uv pip install --system awscurl pyyaml.
Impact
A malicious or broken release could be silently pulled into the CI image.
Remediation
  1. Pin versions: uv pip install --system awscurl==0.33 pyyaml==6.0.2.
  2. Generate a requirements.txt with hashes for reproducibility.
▶ LOW chmod 777 on CI Cache Directories
File
ci/Containerfile:186
Category
Containers — File Permissions
Issue
Cache directories created with chmod -R 777 for OpenShift random-UID compatibility.
Impact
Any process in the container can read/write caches. Low risk given CI isolation.
Remediation
  1. Use chmod -R 775 with chown :0 (group 0) instead — matching zoa-tools/Dockerfile pattern.
▶ LOW Terraform Lock Files Gitignored
File
.gitignore:6
Category
Supply Chain — Infrastructure
Issue
.terraform.lock.hcl in .gitignore prevents provider lock files from being committed. Combined with >= constraints, each terraform init could resolve different versions.
Impact
Low in practice, but developers may run inconsistent provider versions.
Remediation
  1. Remove from .gitignore and commit lock files.
  2. Alternatively, use -lockfile=readonly in CI.

Security Posture

Overall Risk: MEDIUM

Strengths

  • EKS fully private (endpoint_public_access = false) with all audit log types enabled
  • KMS encryption with automatic key rotation on all sensitive resources (RDS, CloudWatch, S3)
  • S3 buckets properly secured: public access blocked, versioning, server-side encryption, lifecycle policies
  • API Gateway uses AWS_IAM authorization (SigV4) — no anonymous access
  • External Secrets Operator for all secret management — zero hardcoded secrets found
  • No IAM wildcard Action: * policies
  • RDS instances: private, encrypted, multi-AZ, with deletion protection
  • GitHub Actions uses SHA-pinned actions (@<commit-hash>)
  • GPG-verified Terraform installation in CI
  • CloudTrail as configurable feature flag (FedRAMP AU-12)
  • zoa-tools image: FIPS crypto policy, non-root user
  • platform-image: non-root user (65534)
  • ArgoCD HA mode with topology spread constraints

Top Priority

Enable VPC Flow Logs and add Kubernetes NetworkPolicies — these are the largest gaps in visibility and lateral movement prevention.
